# 盘中行情源实测结论（美股常规交易时段）

> 实测日期：**2026-07-10**
> 测试时刻：**北京时间 21:46 – 21:53 / 纽约时间 09:46 – 09:53 ET**（开盘后 16–23 分钟，`marketStatus = Open`）
> 方法：腾讯实时快照 `qt.gtimg.cn` + 腾讯当日分时 `web.ifzq.gtimg.cn/.../usMinute` + Nasdaq `api.nasdaq.com/api/quote`，
> 三源**同一次 JS 执行内、同一时刻**取值交叉比对。全部数据经浏览器实测取得，无一条来自推断。

## 一、结论摘要（先看这个）

| 判断项 | 实测结论 |
|---|---|

### 2.2 排除"延时 15 分钟"的反证

若腾讯延时 15 分钟，快照价应等于「15 分钟前分时价」。实测**全部不等**，且差距远超噪声：

- NVDA：快照 205.86，15 分钟前 204.40 → 差 **1.46**（0.71%）
- TSM：快照 435.34，15 分钟前 430.80 → 差 **4.54**（1.05%）
- ASML：快照 1801.95，15 分钟前 1775.58 → 差 **26.37**（1.49%）

同时快照价**逐轮跟随行情推进**（NVDA 快照 09:46→09:53：204.56 → 205.37 → 205.57 → 206.01 → 206.35 → 205.86），
排除"时间戳新但内容是陈旧值"的假象（即新浪 `gb_` 那种坑）。

### 2.3 腾讯分时接口的分钟标签口径

实测发现：**分时 bar 的标签是该分钟的"末刻"**。
例：NY 09:48:34 时，末条 bar 标签为 `0949`。即 `0949` 覆盖 09:48:00–09:48:59，为**正在形成的当前分钟**。
→ 用分时末条当"真实最新价"基准是正确的，但**不要**把标签当成"未来时间"而误判为数据超前。

### 2.4 `live.json` 盘中陈旧实测

| 观测时刻（NY） | `live.json` 的 `t`（NY） | 陈旧度 | `src` | NVDA 报价 | 同刻真实价 |
|---|---|---|---|---|---|
| 09:48:01 | 09:29:14 | **19 分钟** | nasdaq | 201.79 | 205.32 |
| 09:51:06 | 09:29:14 | **22 分钟** | nasdaq | 201.79 | 206.41 |

`t` 始终不动，停在开盘前最后一次写入（09:29:14 = 09:30 开盘前一分钟）。
**盘中前端若使用 `live.json`，拿到的是盘前价，误差 ~2%。**

## 三、明确结论

### 3.1 美股盘中现价该用谁？

**用腾讯（现状即最优，无需改动）。**

- 腾讯快照 `qt.gtimg.cn/q=usXXXX` 第 4 个 `~` 字段 = 实时现价，延时 0–1 分钟；
- 覆盖全，前端可直连（`<script charset="GBK">` 注入），无需经 ECS 转发；
- 分时接口同样实时，可作为校验基准。

**不要**在盘中改用 `live.json`——它盘中不更新。

### 3.2 Nasdaq 对 NYSE 上市标的是否可用？

**可用，且实时。** 这是本次唯一"查资料查不到、只能实测"的点，结论明确：

- Nasdaq 的 `api/quote/{sym}/info` 返回的 `primaryData.lastSalePrice` 对 **NYSE**（TSM、BABA、JPM）
  和 **NYSE Arca / PSE**（SPY）标的**均为实时合并行情**，`isRealTime: true`，时间戳为当前分钟；
- TSM 在 09:52 三源报价 **435.34 / 435.34 / 435.335**，逐分位对齐，不存在"Nasdaq 只有自家成交、对 NYSE 给延时"的问题。
- 常规时段 `data.marketStatus == "Open"`（盘前为 `Pre-Market`，盘后含 `After`）。

**因此 Nasdaq 是一个合格的盘中备份源**（腾讯若哪天挂了/被墙，可无缝切换），但**当前没有必要用它替换腾讯**。

## 四、`live.json` 盘中不生成的根因（已定位到代码行）

**不是 cron 的问题**——`scripts/deploy-ecs-live.sh` 里 `us-live.sh` 是 `* * * * *` 每分钟都在跑。
是**两条取价路径都被显式写死为"只取盘前/盘后"**：

1. **Yahoo 路径** `collect_quotes()`：
   ```python
   if state.startswith("PRE"):    price = q.get("preMarketPrice")
   elif state.startswith("POST"): price = q.get("postMarketPrice")
   if not price or price <= 0: continue      # ← REGULAR 状态直接跳过
   ```
2. **Nasdaq 路径** `fetch_nasdaq_one()`：
   ```python
   if "Pre-Market" not in market_status and "After" not in market_status:
       return symbol, market_status, None, codes, None   # ← marketStatus=="Open" 直接返回空
   ```

两路都返回 0 条 → `len(quotes) < MIN_QUOTES_TO_WRITE` → `LIVE_SKIP ... keep previous live.json`
→ `live.json` 冻结在盘前最后一次写入。**与实测的 `t=09:29:14` 完全吻合。**

这是**设计如此**（当初只为解决盘前/盘后腾讯没数据的问题），不是 bug。

## 五、要不要改造？建议：**不改主链路，只做可选加固**

既然腾讯盘中实时，**改造不是必需的**。若日后仍想让 `live.json` 盘中也可用（作为腾讯的热备），要点如下：

> **铁律：所有 ECS 改动必须落在 `scripts/deploy-ecs-live.sh` 里，禁止手改 `/root/` 下的文件。**

1. **`fetch_nasdaq_one()`**：把交易时段过滤放宽——`marketStatus == "Open"` 时同样取
   `primaryData.lastSalePrice` / `percentageChange`（该接口在 Open 时段返回的就是实时最新成交价，已实测）。
2. **`collect_quotes()`（Yahoo）**：增加 `elif state.startswith("REGULAR"): price = q.get("regularMarketPrice")`，
   涨跌幅取 `regularMarketChangePercent`。
3. **`write_live()`**：在 JSON 里加一个 `sess` 字段（`pre` / `open` / `post`），前端据此判断可用性，
   而不是靠猜时间窗。
4. **前端**：盘中**仍以腾讯为主**，仅当腾讯连续 N 次取值失败时才回落到 `live.json`；
   并且**必须校验 `live.json` 的 `t` 不超过 2 分钟**，否则视为不可用（本次陈旧 22 分钟就是活教训）。
5. 改完后**必须在下一个盘中时段重跑 `scripts/probe-feed-lag.md` 探针复测**，不得凭代码推断。

## 七、方法学备注（供下次复测）

- 腾讯两个接口**都有 CORS 限制**，只能用 `<script>` 标签注入（快照必须 `s.charset='GBK'`）。
- `api.nasdaq.com` **不发 CORS 头**，跨域 `fetch` 必失败。本次的做法是：把浏览器 tab 导航到
  `https://api.nasdaq.com/api/quote/NVDA/info?assetclass=stocks`，**在该 origin 下**执行脚本——
  此时 Nasdaq 是同源可 `fetch`，而腾讯两接口仍可用 `<script>` 注入。**这样才能做到"同一时刻、三源同取"**，
  否则跨调用的时间差会污染比对。
- 沙箱 bash 既连不上腾讯，也连不上 `api.nasdaq.com` 和 `stock.ziyuanai.top`，全程只能靠浏览器。
