> 最后更新：2026-07-10（当日全面实测重写）

## 一、一句话概括

**腾讯管盘中现价，Nasdaq 管美股盘前盘后，CoinGecko 管加密，Yahoo 管所有不怎么变的统计数字。**

四个源各守一段，在页面上拼成一张完整的表。新浪已彻底弃用。

## 二、四个数据源的分工

| 数据源 | 负责什么 | 多快 | 怎么拿到的 |
|---|---|---|---|
| **腾讯** | 盘中现价与当日涨跌（A股 / 美股 / 港股 / 日股 / 韩股 / 台股 + 指数） | 30 秒刷新 | 你的浏览器直连 |
| **Nasdaq** | 美股**盘前 / 盘后**现价与当日涨跌 | 60 秒 | 阿里云 ECS 每分钟抓，写成 `live.json` |
| **CoinGecko** | 加密资产现价、**真实流通市值**、历史高点、24h/7d/30d/1y 涨跌 | 60 秒 | 你的浏览器直连（免 key、可跨域） |
| **Yahoo** | PE、远期PE、财报日、近1周~近1年涨跌、52周区间、历史高点、真实市值 | 15 分钟 | GitHub Actions 流水线，写进 `quotes.json` |

**贯穿始终的分界线：会动的数据走快车道，不会动的走慢车道。**

价格每秒都在变 → 浏览器直连（腾讯 / CoinGecko）或 ECS 直连（Nasdaq），几百毫秒。
PE、52周区间这些一天变不了几次 → 走重型流水线，它要下载 390 个标的的全年 K 线，单次跑约 6 分钟。

早期最大的架构错误，正是让盘前价搭上了那趟慢车。挪到快车道之后，流水线反而可以放慢到 15 分钟，更稳。

## 三、行情延时实测表（2026-07-10）

**免费行情是否延时，取决于交易所授权，不取决于数据源的技术。但具体到每个市场必须实测——网上的说法屡次被推翻。**

| 市场 | 腾讯的延时 | 怎么测出来的 | 状态 |
|---|---|---|---|
| **A股** | 疑似实时 | 资料称 Level-1 免费即实时 | ⏳ **未实测**，周一验 |
| **美股（盘中）** | **实时** | NVDA 快照 205.30 紧贴分时最新 205.57（15分钟前是 203.50）；TSM(纽交所)、SPY 同样实时 | ✅ 已实测 |
| **美股（盘前/盘后）** | 返回昨收（冻结） | 快照价 = 前一交易日收盘价 | ✅ 已实测，故改用 Nasdaq |
| **港股** | **延时 15 分钟** | 13:56 的价，14:11 才从快照接口返回。宁德时代H、时代电气两只精确吻合 | ✅ 已实测 |
| **日股 / 韩股 / 台股** | 未知 | — | ⏳ **未实测**，周一验 |
| **加密（CoinGecko）** | 实时 | 加密无交易所授权约束，行情本就免费实时 | ✅ |

> **测法**（`scripts/probe-feed-lag.md` 有完整脚本）：腾讯有两套接口——实时快照 `qt.gtimg.cn/q=`
> 与当日分时 `web.ifzq.gtimg.cn/appstock/app/...Minute/query`。**把快照返回的价，拿去分时记录里反查它出现在哪一分钟**，
> 与分时最后一条的时间差就是延时。必须在交易时段内做，收盘后所有源都等于收盘价，测不出东西。

**「腾讯美股免费延时10-15分钟」这个网上广为流传的说法，已被本次实测推翻。**
所以美股 189 行盘中不需要切到 Nasdaq——它本来就是实时的。Nasdaq 只在盘前盘后有用。

## 四、前端怎么把四份数据拼起来

- **盘中**：腾讯 30 秒刷新现价与当日涨跌。
- **美股盘前/盘后**（北京 16:00–21:30 / 04:00–08:00）：读 ECS 的 `live.json`（Nasdaq，60 秒）；
  它没覆盖的（OTC、指数）退回 Yahoo 的盘前价。**先快档后慢档**，顺序反了会闪过一个旧价。
- **加密**：CoinGecko 每 60 秒直连，覆盖现价、市值、ATH、24h/7d/30d/1y。
- **休市**：显示上一交易日收盘价与当日涨跌。

页面上还有两个自动推导的量，不等流水线：**市值**（＝现价 × 锚点）和**回撤**（＝现价 ÷ 历史高点）。

## 五、怎么一眼看出价格的性质

- **金色「盘前」角标** → 这是盘前（或盘后）成交价，不是收盘价。无角标的美股行，说明它盘前没有成交，显示的是上一交易日收盘状态（与富途「盘前暂无成交」口径一致）。
- **港股延时 15 分钟** → 不逐行标（行行都有、零信息量），只写在顶部说明里；真要判断买点时，现价悬停的买点浮窗会提示。

## 六、Yahoo 的历史数据有 6 个覆盖缺口（已用腾讯日K补上）

2026-07-10 逐一实测 Yahoo `v8/finance/chart`（日/周/月线、`5y`/`max`/显式起止、query1/query2 结果一致）返回的日线根数：

| 有历史 | 根数 | | 没有历史 | 根数 |
|---|---|---|---|---|
| 上证指数 `000001.SS` | 1211 | | 科创50 `000688.SS` | **1** |
| 沪深300 `000300.SS` | 1211 | | 上证50 `000016.SS` | **1** |
| 深证成指 `399001.SZ` | 1211 | | 中证500 `000905.SS` | **1** |
| 恒生指数 `^HSI` | 1227 | | 中证1000 `000852.SS` | **1** |
| 恒生国企 `^HSCE` | 1227 | | 创业板指 `399006.SZ` | **1** |
| | | | 恒生科技 `HSTECH.HK` | **1**（`firstTradeDate` 亦为 null） |

老牌宽基齐全，这 6 个新指数一律只给 1 根。**是 Yahoo 的覆盖缺口，不是我们的 bug。**

腾讯的日K对这 6 个全都有（1400~1600 根）。故在 `watchlist.json` 里为它们**显式声明**历史源：

```json
{"code": "000688", "yahoo": "000688.SS", "hist": "tx:sh000688"}
```

**这不是「拿不到就偷偷换源」的黑盒兜底**，而是写在配置里一眼可见的声明，只对这 6 只生效；
声明的源若也失败，直接报错，绝不再往下退。区别是本质的：黑盒兜底会把「我们代码写错了」
伪装成「数据源不给力」，让真因永远查不出来。

> 腾讯的日K只覆盖 A股/港股。美股只返回 2 根，日/韩/台各只有 1 根——**别拿它去补别的市场**。

## 七、两个访问入口

- **`https://stock.ziyuanai.top`** —— 阿里云 ECS 托管，**国内直连免翻墙**，秒开。日常用这个。
- `https://chenyanchong321.github.io/us-stock-notes/` —— GitHub Pages，海外备用入口。

内容以 GitHub 仓库为准，ECS 每 2 分钟自动同步。ECS 只额外做一件事：抓 Nasdaq 盘前价。

## 八、自动化（都不需要人工干预）

**ECS 上三个闹钟**：每分钟抓 Nasdaq 写 `live.json`；每 15 分钟触发 GitHub 流水线；每 2 分钟把网站同步到 ECS。

**GitHub 流水线的触发方式**：定时 + 手动 + **改动 `config/**` 或 `scripts/update_quotes.py` 时自动重跑**
（改完配置立刻见结果，不必等闹钟）。提交步骤有 5 次 rebase 重试，checkout 用完整历史
（浅克隆上跑 `git pull --rebase` 会炸，这是曾经 50% 失败率的真凶）。

## 九、三道自动防线（吃过亏才加的）

1. **整批守卫**：`live.json` 里若超过 70% 的报价与收盘价完全相同 → 判定该源在盘前不给盘前价，整批丢弃。
2. **逐只守卫**：某只票的报价恰好等于收盘价 → 跳过该只，不覆盖已取到的正确盘前价。
3. **数据过期告警**：静态数据超时（工作日 8 点后 >1.5 小时）页面自动弹金色横幅，数据恢复后自动收起。
   **每次重拉数据都复查**，页面开着不动也会弹。

所以即使数据源哪天悄悄变坏，页面不会默默显示错价，只会降级并报警。

## 十、已知的盲区与限制

- **美股「夜盘」**（北京白天那段场外交易）：免费源都没有，是富途花钱买的。那正好是你白天忙、最不需要盯盘的时候。
- **Yahoo 从阿里云 ECS 直连会被 403 拦掉**，所以 ECS 上只能用 Nasdaq；Yahoo 只能由 GitHub 流水线抓。
- **Nasdaq 不覆盖 OTC 和指数**，这些标的的盘前价仍靠 Yahoo 流水线。
- **Nasdaq 在个股盘前无成交时**会退回常规时段数据（价=昨收、涨跌幅=昨日的），已由「逐只守卫」拦下。
- **市值为估算口径**（现价 × 锚点推导），非交易所实时口径。跨市场排序时已按汇率折算成美元。
  例外：加密资产用 `mcap_live` 取真实流通市值——稳定币价格恒为 1，锚点推导会把市值永远冻在锚定当天。

## 附一：加密为什么用 CoinGecko（2026-07-10 实测三家）

加密**没有交易所授权那套规矩**，行情本就免费实时。

| | 浏览器直连 | 免 API key | 给市值 |
|---|---|---|---|
| **CoinGecko** | ✅ | ✅ | ✅ 真实流通市值 |
| 币安 | ❌ 被跨域挡 | ✅ | ❌ **没有** |
| CoinMarketCap | ❌ 被跨域挡 | ❌ 需 key | ✅ |

- **币安是交易所，只知成交价、不知流通量，所以给不出市值。** 而稳定币恰恰只能看市值。
- **看 USDT 的市值＝看流通量**：市值扩张是新钱进场，收缩是资金离场。这是加密市场最好用的流动性指标之一。

> 坑：Yahoo 上的 `PEPE-USD` 是另一个币（PEPEGOLD），真正的 Pepe 是 `PEPE24478-USD`。
> 坑：极小价格（PEPE ≈ 0.0000027）用 `toFixed(3)` 会抹成 `0.000`，用 `%.3g` 又会变成科学计数 `2.71e-06`。
> 正确做法是补足小数位再去尾零，且前后端两侧口径必须一致。

## 附二：一个反复用得上的技巧——绕过跨域看原始返回

浏览器不能 `fetch` 别的域，但**可以「导航」到那个域**。一旦页面停在该域下，同源 `fetch` 就全放开了。

排查 Yahoo 时先导航到 `https://query1.finance.yahoo.com/v8/finance/chart/AAPL?range=1d&interval=1m`，
之后在该标签页里可以随意请求任何 Yahoo 接口——包括它自己的搜索接口 `/v1/finance/search?q=`，
用来确认某个标的到底登记成什么代码。

这个技巧终结了「看不到数据源原始返回、只能靠猜」的困境。**猜是今天所有错误的共同根源。**

## 订正记录

### 2026-07-08 那版说明错在哪

写着「新浪管盘前盘后的现价（分钟级）」「雅虎管不怎么变的统计数字」。**两处都错：**

1. **Yahoo 一直支持盘前价**，代码里 `preMarketPrice` 比新浪那层早加了一天多。
2. **新浪根本没有盘前数据**：实测 143 只美股里 137 只的价格与涨跌幅和昨收完全一致。
   更糟的是它每 60 秒用昨收去**覆盖** Yahoo 已取到的正确盘前价，盘前显示不升反降。

新浪那一层的真实作用从来不是「补数据」，而是想「补频率」。把「频率不够」表述成「数据源不支持」，是实打实的错误。

### 2026-07-10 当天犯的两个错

1. **把兜底的功劳记在了自己头上。** 曾断言「指数没历史的根因是盲目优先 `adjclose`」并宣布 5 个指数已修复。
   实际上那批好数据来自上一个还带着腾讯兜底的提交。删掉兜底后它们立刻打回原形，才露馅。
   **教训：静默兜底不但掩盖数据源问题，还会掩盖我们自己的 bug，甚至让人误以为修对了。**
2. **把网上资料当结论。** 曾据「腾讯美股免费延时10-15分钟」的说法，准备把美股 189 行切到 Nasdaq。
   当晚盘中实测发现腾讯美股**根本就是实时的**，整个改造毫无必要。
   **教训：资料只能当线索，结论必须实测。**
